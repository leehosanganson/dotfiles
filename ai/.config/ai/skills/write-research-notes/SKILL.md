## Overview

`$HOME/Documents/research/` is the canonical directory for storing research notes and markdown documentation. All research content should be written here so it persists across sessions and is easily discoverable.

By default, the script creates a timestamped session directory under `research/` named `<auto-generated datetime>-<project>/`. The project name can be specified with the `-p/--project` flag or auto-derived from the topic-slug (first part before `/`, or the full slug if no slash). **All timestamps are fully automated — the script calls `datetime.now()` internally and agents never need to compute or supply any datetime values.**

## Why Write Notes Frequently

**Write research findings as soon as you have them — don't wait until the end.** Every time you:

- Discover a new piece of information
- Complete an analysis step
- Reach a conclusion or hypothesis

…write it down immediately. This matters because:

1. **Discoverability** — Other agents (and your future self) can search and browse `$HOME/Documents/research/` to find what's already been done. Nothing is worse than duplicating work because someone else already solved it but didn't write it down.
2. **Persistence across sessions** — In long research workflows, context gets lost between agent turns. Writing notes after each step creates a durable trail that survives session boundaries.
3. **Avoids duplication** — Always **read existing notes before writing new ones**. Check what's already been documented to avoid repeating effort and to build on prior work.

Research notes are the shared memory of your project. Write often, write early, write clearly.

## How to Use

**Always use `scripts/write-research.py`** bundled with this skill — never write files directly into `$HOME/Documents/research/`. The script handles:

- YAML frontmatter generation (`title`, `date`, `tags`, `status`)
- Timestamped session directory creation (auto-generated by the script; agents never supply a datetime)
- Path resolution and file naming (auto-generated by the script; agents never supply a datetime)
- Content sourcing via `--file`

### The Recommended Workflow (Two Steps)

The recommended workflow is a clean two-step process: write content to a temp file, then run the script pointing to it.

```bash
# Step 1: Write content to /tmp/content.md (using your Write tool)
# Step 2: Run the script with --file
uv run scripts/write-research.py topic-slug --file /tmp/content.md
# → writes to ~/Documents/research/20260523_143000-topic-slug/20260523_143000-topic-slug.md
```

This pattern is clean, reliable, and works in all agent contexts. The Write tool handles the content creation; the script handles frontmatter, directory structure, and file naming. All timestamp generation is fully automated — you never need to compute or provide any datetime values.

### Key Rules

1. **Use `--file` with a temp file** — This is the only supported pattern for providing content. Write your content first (e.g., with your Write tool into `/tmp/content.md`), then point the script at it. No inline strings, no pipes, no heredocs needed in the command itself.
2. **Use `-t` for explicit title** — Optional; auto-derived from slug if omitted. Always provide it when the slug alone doesn't make a good title.
3. **Use `-p/--project` to specify a project name** — Determines the session subdirectory (`<datetime>-<project>/`). If not given, auto-derived from topic-slug (first part before `/`, or full slug). The timestamp portion is always auto-generated; never supply it yourself.
4. **Use `-f/--filepath` for explicit output path** — Overrides auto-resolution entirely, bypassing timestamped directory creation. Use when you need a specific location outside the session directory.

### Full Workflow Examples

```bash
# Example 1: Simple note — auto-derives project from slug, creates session dir
# Step 1: Write content to temp file (using your Write tool)
# (Agent writes the following into /tmp/content.md using the Write tool)

# Step 2: Run the script
uv run scripts/write-research.py proxmox-server-build --file /tmp/content.md
# → ~/Documents/research/20260523_143000-proxmox-server-build/20260523_143000-proxmox-server-build.md

# Example 2: With explicit project override
# Step 1: Write content to temp file
# (Agent writes into /tmp/content.md)

# Step 2: Run with -p flag
uv run scripts/write-research.py db-migration -p k8s-setup --file /tmp/content.md
# → ~/Documents/research/20260523_143000-k8s-setup/20260523_143000-db-migration.md

# Example 3: Multiple notes under the same session directory (write each one separately)
# Note 1 — write content to /tmp/content.md, then run:
uv run scripts/write-research.py findings -t "Cluster Findings" -p my-project --file /tmp/content.md
# → ~/Documents/research/20260523_143000-my-project/20260523_143000-findings.md

# Note 2 — update /tmp/content.md with new content, then run again:
uv run scripts/write-research.py summary -t "Summary Report" -p my-project --file /tmp/content.md
# → ~/Documents/research/20260523_143000-my-project/20260523_143000-summary.md

# Example 4: Explicit filepath override (bypasses session directory)
uv run scripts/write-research.py dl-overview -t "Deep Learning Overview" \
  -f ~/Documents/research/dl-overview.md --file /tmp/content.md
# → ~/Documents/research/dl-overview.md
```

### Research Workflow — Write at Every Step

A good research session writes notes incrementally. Here's the pattern:

```bash
# Step 1: Read existing notes first to avoid duplication
# (Use your file reading tools to browse $HOME/Documents/research/)

# Step 2: Write initial findings as soon as you discover them
uv run scripts/write-research.py hypothesis -t "Initial Hypothesis" -p my-project --file /tmp/content.md

# Step 3: Write intermediate results after each analysis step
uv run scripts/write-research.py analysis-results -t "Analysis Results" -p my-project --file /tmp/content.md

# Step 4: Write conclusions when you reach them
uv run scripts/write-research.py conclusions -t "Conclusions" -p my-project --file /tmp/content.md
```

**Don't batch everything into one final write.** Each individual note is self-contained and discoverable. Breaking notes into smaller, focused files makes them easier to search, reference, and update.

## File Structure

The script auto-organizes notes into timestamped session directories. The timestamps shown below are examples — in practice they are always generated automatically by the script at runtime:

```
~/Documents/research/
├── 20260523_143000-k8s-setup/
│   └── 20260523_143000-k8s-setup.md
└── 20260523_143000-my-project/
    ├── 20260523_143000-findings.md
    ├── 20260523_143000-analysis-results.md
    └── 20260523_143000-conclusions.md
```

The script also generates YAML frontmatter automatically:

```yaml
---
title: Your Title
date: 2026-05-23
tags: []
status: draft
---
```

Your markdown content follows the frontmatter. The `tags` and `status` fields are empty by default — edit them after writing if needed.

## Reading Existing Notes

Before writing new notes, **read existing ones** from `$HOME/Documents/research/` to avoid duplication and build on prior work. Use your file reading tools to browse the directory structure and read relevant files.

```bash
# Browse the research directory structure
ls ~/Documents/research/

# Read specific notes for reference
# (Use your file reading tool — no bash needed)
```

Always check what's already documented before writing new content. This prevents wasted effort and ensures continuity across research sessions.
